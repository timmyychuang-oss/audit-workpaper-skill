# 质量验证关卡 / Quality Gates

## 概述

质量验证分三层：
1. **阶段内验证**：每个阶段输出前的格式和逻辑检查
2. **跨阶段一致性**：S6输出前的全局检查
3. **全链路追溯**：随机抽样验证推导链完整性

---

## 一、阶段内验证

### S1 总览验证

| 检查项 | 通过条件 |
|--------|---------|
| 战略目标数量 | 3-5个 |
| 目标可反推风险 | 每个目标能回答"什么会失败" |
| 目标无包含关系 | 目标之间独立（如有包含，拆分） |
| 审计范围边界 | 明确"做什么+不做什么" |
| 底线类型已定义 | 至少2种类型 |

### S2 底线风险验证

详见 `blr_12col.md` 的"12列质量验证清单"。额外全局检查：

| 检查项 | 通过条件 |
|--------|---------|
| BLR总数 | ≥ 战略目标数 × 1.5 |
| 每个目标覆盖 | 每个战略目标至少1个BLR关联 |
| 底线类型分布 | 不应全部集中在一种类型 |
| P级初判 | 每个BLR有初步P级判断 |

### S3 测试方案验证

详见 `test_7col.md` 的"7列质量验证清单"。额外全局检查：

| 检查项 | 通过条件 |
|--------|---------|
| 测试总数 | BLR数 × 1.5 ~ 2.5（合理区间） |
| BLR覆盖率 | 每个BLR至少1个测试 |
| 证据类型分布 | 不应全部是数据分析型（除非确实全是数据审计） |
| 列12已回填 | 所有BLR的补偿性控制"验证方向"已填入测试编号 |

### S4 证据源验证

| 检查项 | 通过条件 |
|--------|---------|
| 测试覆盖 | 每个测试至少1个证据源 |
| 无孤立证据源 | 每个证据源有审计用途（测试编号） |
| 来源分类完整 | 每个证据源有来源分类 |
| 外部方独立性 | 被审方/供应商提供的证据有独立验证方法 |
| 数据缺口标注 | 不可获取的证据标注"数据缺口"+替代方案 |

### S5 待办验证

| 检查项 | 通过条件 |
|--------|---------|
| 测试全覆盖 | 每个测试在待办中有对应项 |
| P级一致性 | 待办P级 ≥ 对应BLR最高P级 |
| 证据获取排序 | P0证据获取排在最前 |
| 分析任务→证据映射 | 每个分析任务的"所需证据"在证据获取待办中有对应项 |

## 二、跨阶段一致性检查（S6输出前执行）

在生成最终Excel前，执行以下8项全局检查。**任何一项不通过，必须修复后再输出。**

### 检查1: 目标→BLR覆盖
```
遍历S1的每个战略目标：
  该目标是否被至少1个BLR的列4引用？
  如果否 → 追问用户"目标X没有关联的底线风险，是否遗漏？"
```

### 检查2: BLR→测试覆盖
```
遍历S2的每个BLR：
  该BLR是否被至少1个测试的列G引用？
  如果否 → 提示"BLR-N没有对应测试，需要补充设计"
```

### 检查3: 测试→证据源覆盖
```
遍历S3的每个测试：
  该测试是否在Sheet 4中有至少1个证据源？
  如果否 → 提示"测试T-N.M没有证据源，需要补充映射"
```

### 检查4: 证据源→测试反向映射
```
遍历Sheet 4的每个证据源：
  该证据源的"审计用途"列是否有测试编号？
  如果否 → 删除该证据源（无用证据不应出现在底稿中）
```

### 检查5: 外部方数据独立性
```
遍历Sheet 5中来源为"被审方提供"或"供应商提供"的证据源：
  是否有"独立验证方法"？
  如果否 → 标注为"⚠ 对账盲区"，在Sheet 6中增加待办项"建立独立验证机制"
```

### 检查6: P级一致性
```
遍历Sheet 6的每个待办项：
  待办P级 ≥ 其关联BLR中最高的P级？
  如果否 → 调整待办P级
```

### 检查7: 列12回填
```
遍历S2的每个BLR列12：
  "验证方向"是否已从描述文字更新为测试编号（如"→ 见T1.2"）？
  如果否 → 执行回填
```

### 检查8: 全链路追溯
```
随机抽取1个Sheet 6待办项：
  待办 → 对应测试编号 → 对应BLR编号 → 对应战略目标
  链路是否完整？每一步引用是否有效？
  如果链路断裂 → 修复断链
```

## 三、全链路追溯验证方法

### 正向追溯（从目标到待办）
```
战略目标 "快速铺设"
  → BLR-1 "铺设节奏失控（需求与供应失配）"
    → T1.1 "需求预测准确率分析"
      → 证据源：报表1（需求计划明细表）
        → 待办P0-1："获取报表1"
          → 待办P0-A1："执行T1.1分析"
```

### 反向追溯（从待办到目标）
```
待办P0-A1 "执行T1.1分析"
  → T1.1 "需求预测准确率分析"
    → BLR-1 "铺设节奏失控"
      → 战略目标 "快速铺设"
✓ 链路完整
```

如果任何一步引用断裂（如测试T1.1不存在，或BLR-1没有关联目标），则链路不通过。

---

## 四、一致性修复流程

当检查发现不一致时：

1. **向用户报告**不一致项，说明具体问题
2. **提出修复建议**（补充遗漏 / 调整P级 / 删除孤立项）
3. **用户确认**修复方案
4. **执行修复**
5. **重新运行**受影响的检查项
6. 全部通过后才输出Excel

---

## 五、Excel输出后的最终检查

Excel生成后，做最后一轮人工可读性检查：

| 检查项 | 方法 |
|--------|------|
| Sheet 1 总览 | 确认审计范围、目标清单显示完整 |
| Sheet 2 底线风险 | 确认12列全部有内容，无空列 |
| Sheet 3 测试方案 | 确认7列全部有内容，步骤编号连续 |
| Sheet 4 证据源清单 | 确认每行有审计用途，无孤立行 |
| Sheet 5 来源分类 | 确认外部方数据有独立验证方法或标注盲区 |
| Sheet 6 待办 | 确认P0项排在最前，每项有对应测试编号 |
