# 测试方案7列模型 / Test Plan 7-Column Model

## 推导逻辑

对每个已确认的BLR，问：**"怎么用证据证明这个风险存不存在？"**

每个BLR拆解为1-3个测试（T编号），用以下7列展开。

---

## 7列定义

### 列A: 审计目标
- **第一句必须是问句**（以？结尾）
- 用口语化表达，让初级审计人员一读就懂
- 问句后解释为什么这个问题重要

**写法范例：**
```
配件需求预测到底准不准？

具体来说：业务团队提交的月度需求计划（"下个月需要多少指纹模组"），
和实际的采购申请量之间，差距有多大？
如果预测系统性偏高或偏低，意味着资金效率和交付节奏同时失控。
```

**常见错误：**
- 不是问句："验证需求预测准确率"（应改为"需求预测到底准不准？"）
- 太术语化："评估MAPE指标合规性"（应改为白话）

### 列B: 证据源
- 精确到具体的表名/文件名/访谈对象
- 每个测试至少引用1个证据源
- 区分主要证据源和辅助证据源

**写法范例：**
```
主要：报表1（需求计划明细表）+ 报表3（业务申请明细表）
辅助：维表1（需求预测维表）+ 维表14（物料需求计划明细表）
```

### 列C: 分析步骤
- 每步以**"第N步："**开头
- 每步 = 一个可执行的单一动作
- ≥5步
- **必须包含"交叉验证"步骤**（至少与一个独立证据源比对）

根据测试性质，选择以下三种模板之一：

---

#### 模板A：数据分析型

适用于：有结构化数据可用的测试（数据库、报表、日志）

```
第一步：从{表A}提取{字段X}。
       说明提取条件（时间范围、筛选规则）。

第二步：从{表B}提取{字段Y}作为对比基准。
       说明为什么这个基准可信。

第三步：计算{指标} = {公式}。
       写出完整公式，不省略。

第四步：按{维度}分组分析（如按月/按供应商/按品类），
       画{图表类型}（折线图/柱状图/帕累托图）。

第五步：识别{异常模式}——
       定义什么算"异常"（如MAPE突然飙高的月份）。

第六步：交叉验证——
       与{独立数据源}比对，验证发现是否一致。

第七步：追问异常原因——
       对异常数据下钻到{具体维度}，
       去对应发生了什么业务事件。
```

#### 模板B：文档审阅型

适用于：合同审查、制度审阅、合规检查

```
第一步：调阅{文件/合同/制度名称}，
       确认版本和生效日期。

第二步：建立检查清单——列出需检查的关键条款/要素，
       来源于{法规/标准/最佳实践}。

第三步：逐项检查——对每个条款标注：
       ✓有且完整 / △有但不完整 / ✗缺失。

第四步：对缺失项和不完整项评估影响——
       缺失意味着什么风险？影响范围多大？

第五步：交叉验证——与实际执行情况比对，
       "写了的有没有在做？做了的有没有写？"

第六步：形成gap清单——按严重程度排序，
       标注每个gap的风险等级和建议改善措施。
```

#### 模板C：访谈观察型

适用于：流程审计、人员访谈、现场观察

```
第一步：确定访谈对象——{角色/职责}，
       解释为什么选这个人（掌握什么信息）。

第二步：围绕{BLR相关问题}设计访谈提纲，
       每个问题聚焦一个具体事实，不问观点。

第三步：记录关键回答——聚焦事实而非观点，
       对每个回答标注"可验证"或"仅为口述"。

第四步：交叉验证——同一问题问{另一个角色}，
       比对答案一致性，标注矛盾点。

第五步：与文档/数据比对——
       口述内容 vs 书面记录/系统数据，
       标注一致或差异。

第六步：整理发现——区分：
       ① 有多方证据支持的事实
       ② 仅有单方口述的待验证信息
       ③ 存在矛盾的待调查信息
```

---

### 列D: 预期输出
- 列出≥3项交付物
- 至少包含1个可视化（图表）+ 1个清单/表格
- 明确输出的具体内容

**写法范例：**
```
月度MAPE趋势图（折线图）
Bias趋势图（判断系统性偏差方向）
分物料的预测准确率排名
异常月份清单及可能原因
```

### 列E: 判断标准
- **分级阈值格式**：`{条件} = {结论}`
- 必须有≥2级阈值（一般问题 + 严重问题）
- 必须有≥1个模式识别规则

**写法范例：**
```
MAPE > 30% = 预测体系需要系统性改进          ← 一般问题（黄线）
MAPE > 50% = 预测几乎没有参考价值            ← 严重问题（红线）
Bias持续同方向 > 3个月 = 存在系统性偏差        ← 模式识别
某物料MAPE显著高于均值 = 该物料需专项关注       ← 模式识别
```

**常见错误：**
- 没有分级："MAPE不达标需要改进"（缺少具体阈值和分级）
- 只有一级：只写了红线没写黄线
- 纯定性描述："质量较差"（应量化）

### 列F: 证据支撑度
- ★~★★★★★评级
- 评估维度：数据是否存在？是否完整？是否可获取？

| 评级 | 含义 | 行动 |
|------|------|------|
| ★★★★★ | 数据充分，可直接分析 | 正常执行 |
| ★★★★ | 数据基本有，需确认某个字段/环节 | 标注需确认项 |
| ★★★ | 依赖多个数据源交叉，需验证可行性 | 标注依赖关系 |
| ★★ | 非数据分析类，需访谈/调阅文档 | **必须附替代方案** |
| ★ | 数据缺口（如手工台账、未数字化） | **必须附替代方案+标注风险** |

### 列G: 关联BLR
- 格式：`BLR-{N}`
- 每个测试必须关联至少1个BLR
- 一个测试可关联多个BLR
- **反向检查：每个BLR至少被1个测试覆盖**

---

## 测试编号规则

- 格式：`T{BLR编号}.{序号}`
- 例：BLR-1的第2个测试 = `T1.2`
- 如果一个测试同时服务多个BLR，编号取主要BLR

---

## 7列质量验证清单

在展示给用户确认前，逐条检查：

| # | 检查项 | 通过条件 | 不合格处理 |
|---|--------|---------|----------|
| 1 | 审计目标是问句 | 第一句以？结尾 | 改为问句 |
| 2 | 证据源具体 | 有表名/文件名/访谈对象 | 补具体名称 |
| 3 | 步骤≥5且有编号 | "第N步"格式 | 补步骤 |
| 4 | 包含交叉验证步骤 | 至少1步引用独立证据源 | 补交叉验证 |
| 5 | 预期输出≥3项 | 含图表+清单 | 补输出 |
| 6 | 判断标准≥2级 | 黄线+红线 | 补级别 |
| 7 | 有模式识别规则 | ≥1条 | 补规则 |
| 8 | 证据支撑度有评级 | ★~★★★★★ | 补评级 |
| 9 | ≤★★有替代方案 | 附替代获取方式 | 补方案 |
| 10 | 关联BLR有效 | BLR编号存在 | 修正 |
| 全局 | 每个BLR≥1个测试 | — | 补设计 |
| 全局 | 测试总数合理 | BLR数×1.5~2.5 | 追问遗漏或过多 |
