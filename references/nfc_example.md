## 一、总览范例（Sheet 1）

```
审计对象：碰一下NFC线圈供应链
业务描述：支付宝"碰一下"功能的NFC线圈硬件采购、生产、配送、铺设全链路
业务阶段：快速扩张期
审计范围：从需求计划到终端铺设的供应链全流程

战略目标：
1. 快速铺设：按计划完成全国商户覆盖
2. 终端体验：线圈质量保障用户碰一下成功率
3. 成本控制：供应链成本在预算范围内
4. 供应连续性：不因供应中断影响铺设进度

底线类型：战略底线 / 财务底线
```

## 二、底线风险12列范例（Sheet 2）

### BLR-1 铺设节奏失控（需求与供应失配）

| 列 | 内容 |
|----|------|
| 1 编号 | BLR-1 |
| 2 名称 | 铺设节奏失控（需求与供应失配） |
| 3 底线类型 | 战略底线 |
| 4 关联目标 | 快速铺设 + 成本控制 |
| 5 风险描述 | 碰一下的需求预测到底准不准？如果预测系统性偏高，库存积压占用资金；如果系统性偏低，紧急采购增加成本15-30%。真正的痛点不是预测不准本身，而是预测偏差没有被及时发现和纠正，导致铺设计划和资金效率同时受损。 |
| 6 逻辑链 | 需求预测偏差大 → 紧急采购增加（成本上升15-30%）或库存积压（资金占用） → 铺设计划延迟或资源浪费 → 市场覆盖目标无法按时达成 / 资金效率下降 |
| 7 底线标准 | 需求预测MAPE ≤ 30% [L2: APICS基准] &#124; 紧急订单占比 ≤ 15% [L4: 被审方KPI] &#124; 呆滞库存(>90天)占比 ≤ 5% [L5: 专业判断] &#124; 库存周转天数 ≤ 30天 [L5: 专业判断] |
| 8 失控后果 | 假设月采购额￥500万：紧急订单溢价15% × 超标部分20% × 12月 = ￥180万/年；呆滞库存5%超标 → 资金沉淀￥300万 |
| 9 行业先例 | 某消费电子公司NFC天线预测偏差导致3个月库存积压，报废损失￥800万 [来源：行业访谈，待确认] |
| 10 行业水位+合规 | 行业水位：成熟供应链MAPE通常15-25%，碰一下处于扩张期，30%为合理底线。合规定位：最佳实践（非强制要求） |
| 11 替代方案 | 最基础（零成本）：建立月度需求vs实际复盘会议 &#124; 进阶：统计预测模型+偏差预警 &#124; 标杆：AI驱动需求感知系统 |
| 12 补偿控制 | ① 采购团队经验补救（→ 见T1.2紧急订单分析） ② 供应商弹性产能缓冲（→ 见T2.2产能利用率分析） |

### BLR-3 结算资金损失（对账机制与数据一致性缺陷）

| 列 | 内容 |
|----|------|
| 1 编号 | BLR-3 |
| 2 名称 | 结算资金损失（对账机制与数据一致性缺陷） |
| 3 底线类型 | 财务底线 |
| 4 关联目标 | 成本控制 |
| 5 风险描述 | 对账机制中某个检查点缺失（如结算数量未与收货确认数量核对，或辅料单价无独立基准比对），低值品类单笔差异小不引起关注，但百万/千万级数量放大差异后，累积资金损失可能达百万至千万级。 |
| 6 逻辑链 | 对账机制某检查点缺失 → 数量/价格差异未被发现 → 低值品类单笔差异小不引起关注 → 百万/千万级数量放大差异 → 累积资金损失达百万至千万级 → 成本控制目标落空 |
| 7 底线标准 | 结算数量与收货确认数量一致率 = 100% [L1: 财务基本要求] &#124; 结算单价与合同单价一致率 = 100% [L1] &#124; 全链路守恒差异 < 0.5% [L5: 专业判断] &#124; 三单匹配率 ≥ 99% [L5] &#124; 辅料溢价率 < 15% [L5] |
| 8 失控后果 | 假设月结算额￥800万：0.5%守恒差异 × 12月 = ￥48万/年；辅料溢价15%超标部分 × 辅料占比30% × 年结算额 = 数百万级 |
| 9 行业先例 | 制造业常见：某代工厂因辅料用量未独立验证，年度多付超￥200万 [来源：审计行业案例，待确认] |
| 10 行业水位+合规 | 行业水位：成熟企业三单匹配率>99.5%。合规定位：结算一致性=基本要求（财务审计必查项） |
| 11 替代方案 | 最基础（零成本）：在现有Excel对账模板中增加收货确认数量比对列 &#124; 进阶：自动化三单匹配+差异预警 &#124; 标杆：ERP系统内置全链路数量守恒校验 |
| 12 补偿控制 | ① 供应商信誉和长期合作关系降低故意差错概率（→ 见T3.1结算数量验证） ② 财务月度对账可能捕获大额差异（→ 见T3.3三单匹配） |

## 三、测试方案7列范例（Sheet 3）

### T1.1 需求预测准确率分析

| 列 | 内容 |
|----|------|
| A 审计目标 | 碰一下的需求预测到底准不准？具体来说：业务团队提交的月度需求计划，和实际的采购申请量/铺设量之间，差距有多大？ |
| B 证据源 | 主要：报表1（需求计划明细表）+ 报表3（业务申请明细表）。辅助：维表1 + 维表14 |
| C 分析步骤 | 第一步：从报表1提取每月每种物料的需求预测数量。第二步：从报表3提取同月同物料的实际申请数量。第三步：计算预测偏差 = (预测量-实际量)/实际量 × 100%。第四步：计算MAPE，按月趋势画图。第五步：计算Bias方向——持续为正=系统性高估，持续为负=系统性低估。第六步：按物料类型分组——是所有物料都偏差大，还是特定物料预测特别不准？第七步：识别异常月份——MAPE突然飙高的月份，对应那个月发生了什么业务事件。 |
| D 预期输出 | 月度MAPE趋势图（折线图）&#124; Bias趋势图 &#124; 分物料预测准确率排名 &#124; 异常月份清单及可能原因 |
| E 判断标准 | MAPE > 30% = 预测体系需系统性改进 &#124; MAPE > 50% = 预测几乎没有参考价值 &#124; Bias持续同方向 > 3个月 = 存在系统性偏差 &#124; 某物料MAPE显著高于均值 = 该物料需专项关注 |
| F 证据支撑度 | ★★★★★ 数据充分 |
| G 关联BLR | BLR-1 |

### T3.1 结算数量一致性验证

| 列 | 内容 |
|----|------|
| A 审计目标 | 结算数量是否以我方确认的收货数量为准？这是对账机制最关键的检查点：结算的事实基础是"我方确认收到了多少"，而不是"供应商声称发了多少"。 |
| B 证据源 | 主要：维表3（生产费用已结算）+ ADM宽表（收货确认数量）。辅助：维表5/6（单价基准） |
| C 分析步骤 | 第一步：从维表3提取每笔结算的结算数量和金额。第二步：从ADM宽表提取对应订单的我方收货确认数量。第三步：逐单比对：结算数量 vs 收货确认数量。差异A：结算>收货=多付风险。差异B：结算<收货=少付。第四步：计算总体差异率 = 差异数量之和/总结算数量 × 100%。第五步：按供应商分组——哪个差异最大？是否系统性偏差？第六步：追溯差异原因——差异来自发货环节还是结算环节？第七步：评估收货确认机制——我方如何确认收货？逐件清点还是按发货单签收？ |
| D 预期输出 | 逐单一致性比对清单 &#124; 差异率（按金额/按数量）&#124; 按供应商差异分布 &#124; 多付风险金额汇总 &#124; 收货确认机制评估意见 |
| E 判断标准 | 结算≠收货确认 = 对账缺陷（不论差异率大小）&#124; 差异率 > 0.5% = 系统性问题 &#124; 多付风险 > ￥10万/月 = 需升级汇报 &#124; 收货方式为"按发货单签收" = 机制本身需改进 |
| F 证据支撑度 | ★★★★★ 数据充分 |
| G 关联BLR | BLR-3 |
