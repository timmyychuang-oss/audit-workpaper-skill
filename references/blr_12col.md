# 底线风险12列模型 / Bottom-Line Risk 12-Column Model

## 推导逻辑

对每个已确认的战略目标，问：**"什么会让这个目标失败？"**

每个失败模式 = 1个底线风险（BLR），用以下12列展开。

---

## 12列定义

### 列1: 编号
- 格式：`BLR-{N}`（N从1开始递增）
- 自动生成，不重复

### 列2: 名称
- 格式：`{现象}（{本质}）`
- 约束：≤20字
- 好例子：`铺设节奏失控（需求与供应失配）`
- 坏例子：`供应链可能存在一些问题`（太模糊）

### 列3: 底线类型
- 从用户在S1确认的类型中选择
- 常见选项：战略底线 / 财务底线 / 合规底线 / 安全底线 / 运营底线 / 隐私底线
- 用户可自定义

### 列4: 关联业务目标
- 必须引用S1中用户已确认的战略目标，原文引用
- 不能引用未确认的目标
- 一个BLR可关联多个目标

### 列5: 风险描述
- **三段式结构**：
  1. 现状是什么（客观事实）
  2. 如果A发生 / 如果B发生（场景展开）
  3. 真正的痛点是什么（对业务的实质影响）
- 必须包含"如果"场景展开
- 用白话写，不用术语堆砌

**写法范例：**
```
碰一下的需求预测到底准不准？如果预测系统性偏高，
库存积压占用资金；如果系统性偏低，紧急采购增加成本15-30%。
真正的痛点不是预测不准本身，而是预测偏差没有被及时发现和纠正，
导致铺设计划和资金效率同时受损。
```

### 列6: 风险逻辑链
- **格式**：`A → B → C → D`（用→连接）
- **约束**：
  - ≥3步
  - 起点 = 可控/可观测的条件（不能是天灾、黑天鹅）
  - 终点 = 战略目标受损（必须回到列4的目标）
  - 每步之间有因果关系，不跳步

**写法范例：**
```
需求预测偏差大
→ 紧急采购增加（成本上升15-30%）或库存积压（资金占用）
→ 铺设计划延迟或资源浪费
→ 市场覆盖目标无法按时达成 / 资金效率下降
```

**常见错误：**
- 跳步：`A → D`（中间缺少B和C）
- 起点不可控：`地震 → 供应链中断`
- 终点不回到目标：`成本上升 → 利润下降`（如果战略目标不包含利润，则断链）

### 列7: 底线标准
- **格式**：`{指标名} {运算符} {阈值}（{超出含义}）`
- 每个BLR至少3个量化指标
- **每个阈值必须标注来源层级**

#### 阈值来源5级分类

| 级别 | 来源 | 可信度 | 标注格式 |
|------|------|-------|---------|
| L1 | 法规/监管硬性要求 | 最高 | `[来源：{法规名}§{条}]` |
| L2 | 行业标准/协会基准 | 高 | `[来源：{标准名}，{年份}]` |
| L3 | 头部企业公开实践 | 中高 | `[来源：{公司}公开数据]` |
| L4 | 被审方自设KPI | 中 | `[来源：被审方{文件名}]` |
| L5 | 审计师专业判断 | 低 | `[来源：专业判断，建议审计中校准]` |

**只有L5的阈值，必须在S3测试方案中设计"校准步骤"。**

**写法范例：**
```
需求预测MAPE ≤ 30%（超出=预测体系需系统性改进）
  [来源：供应链管理协会APICS基准，L2]
紧急订单占比 ≤ 15%（超出=需求计划系统性失效）
  [来源：被审方年度KPI，L4]
呆滞库存（>90天）占比 ≤ 5%
  [来源：专业判断，建议审计中校准，L5]
```

### 列8: 失控后果与影响量级
- **必须有金额估算**（¥或$）
- **必须展示计算过程**：基数 × 比率 = 金额

#### 金额估算3级降级

| 情况 | 处理方式 |
|------|---------|
| 用户能给基数 | 直接计算：`月采购额￥500万 × 紧急订单溢价15% × 12月 = ￥900万/年` |
| 用户不确定基数 | 假设性估算：`假设月采购额在￥300-800万区间，年溢价损失￥540-1440万` |
| 无法估算 | 描述维度+规模级别：`影响全量供应商结算，规模为百万至千万级` + 标注"待审计中量化" |

### 列9: 行业先例
- **格式**：`{公司/事件} → {发生了什么} → {后果+金额}`
- 必须联网搜索验证
- 标注来源URL或"待用户确认"
- **搜不到就写"该领域缺乏公开先例"，绝不编造**

**写法范例：**
```
宁夏泾源县2025年7月执法：药品销售时间9:33 vs 处方审核时间9:51，
处方晚于销售18分钟，罚款7,000元
[来源：新浪财经2025-11-08引宁夏市场监管局]
```

### 列10: 行业水位与合规定位
- **两段式结构**：
  1. 行业水位：同行业的普遍做法和水平（用数据说话）
  2. 合规定位：标注强制程度

#### 合规强制程度四级分类

| 级别 | 含义 | 审计含义 |
|------|------|---------|
| 强制 | 法规明文要求，违反=处罚 | 必须合规，无讨论空间 |
| 基本要求 | 行业监管期望，不合规=被关注 | 应该合规 |
| 最佳实践 | 头部企业做法，不做=落后 | 建议参考 |
| 增值建议 | 审计增值，做了更好 | 可选 |

### 列11: 管理替代方案
- **三层递进结构**：
  1. 最基础（成本≈0）：不需要投入就能做的改善
  2. 进阶：需要一定投入的标准做法
  3. 标杆：行业领先企业的做法
- 最基础层必须标注"低成本"或"零成本"

**写法范例：**
```
最基础（零成本）：建立月度需求vs实际复盘会议，追踪MAPE趋势
进阶：引入统计预测模型辅助人工判断，设置预测偏差自动预警
标杆（华为/苹果做法）：AI驱动的需求感知系统，实时调整预测
```

### 列12: 补偿性控制预判
- 回答：**"如果这个风险目前没有爆发，可能是什么在拦截它？"**
- 列≥2个可能的补偿性控制
- 每个补偿性控制附"验证方向"

**S2阶段**：验证方向先写描述文字
**S3完成后**：回填为具体测试编号（如"→ 见T1.2"）

**写法范例：**
```
可能的补偿性控制：
① 采购团队的经验判断在一定程度上弥补了预测不足
   → 验证方向：分析紧急订单中"经验补救"vs"系统失效"的比例（见T1.2）
② 供应商有一定的弹性产能缓冲
   → 验证方向：检查供应商产能利用率是否在70-85%合理区间（见T2.2）
```

## 12列质量验证清单

在展示给用户确认前，逐条检查：

| # | 检查项 | 通过条件 | 不合格处理 |
|---|--------|---------|----------|
| 1 | 编号唯一 | BLR-N不重复 | 重新编号 |
| 2 | 名称格式 | "{现象}（{本质}）"，≤20字 | 精简 |
| 3 | 底线类型有效 | 在已定义类型中 | 选择或自定义 |
| 4 | 关联目标有效 | 引用S1确认的目标 | 修正引用 |
| 5 | 风险描述含"如果" | 包含场景展开 | 补充场景 |
| 6 | 逻辑链≥3步且闭合 | 起点可控+终点回到目标 | 补链或修正 |
| 7 | 底线标准≥3个+来源 | 每个有阈值+来源层级 | 补指标或搜索 |
| 8 | 失控后果有金额 | 有¥/$计算过程 | 要求基数或降级 |
| 9 | 行业先例真实 | 联网验证或标注"待确认" | 搜索或标注 |
| 10 | 合规定位有级别 | 四级之一 | 补标注 |
| 11 | 替代方案三层 | 基础+进阶+标杆 | 补层 |
| 12 | 补偿控制≥2个 | 每个有验证方向 | 补充 |
| 全局 | BLR数≥目标数×1.5 | — | 追问遗漏 |
| 全局 | 每个目标≥1个BLR | — | 补推导 |
